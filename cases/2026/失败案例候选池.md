# 失败案例候选池

> 版本：v0.2 · 2026-05-08
> 用途：为 SK 产品拆解补充反例样本，避免幸存者偏差
> 边界：本文件不是失败八卦库，只记录能映射 failure_modes、能服务产品判断的失败案例
> 关联文件：
> - `core/failure_modes.yml`
> - `cases/2026/case-cards.md`
> - `cases/2026/产品对标库-扩容候选池.md`
> - `radar/signals.md`

---

## 一、为什么需要失败案例池

只拆成功案例会导致幸存者偏差。

失败案例池用于回答：

1. 这个产品为什么没有成功？
2. 它命中了哪个 failure_mode？
3. 它和哪个成功案例形成对照？
4. 它能帮助我们排除什么方向？
5. 它对中国复制判断有什么提醒？

## 三、状态标签

| 状态 | 含义 |
|---|---|
| dead | 已关停 / 破产 / 停止服务 |
| acquired-low | 低价收购 / 失败式退出 |
| shrinking | 明显收缩 / 多地闭店 / 用户流失 |
| pivoted | 原模式失败后转型 |
| zombie | 仍存在但增长逻辑基本失效 |
| disputed | 失败迹象明显，但证据不足 |

## 五、候选进入规则

候选案例进入本文件，至少满足 3 条：

1. 有明确失败 / 收缩 / 转型证据；
2. 有公开 URL 或可核对来源；
3. 能对应 1–3 个 `failure_modes.yml`；
4. 能作为某个成功案例的对照组；
5. 对中国机会判断有帮助；
6. 不是单纯八卦，而能抽象出机制。

## 七、失败案例来源

### F1：失败案例库 / Post-mortem 数据库

- CB Insights startup failure post-mortems
- Failory Startup Cemetery / Graveyard
- 创始人关停信
- 投资人复盘
- 媒体调查报道

### F2：成功案例的失败对照组

每拆一个成功产品，至少找一个失败对照。

示例：

| 成功产品   | 失败对照                            |
| ---------- | ----------------------------------- |
| Gamma      | Tome                                |
| Lovable    | Builder.ai                          |
| Paid       | Zendesk 结果计费争议 / Leena AI     |
| ListenLabs | 失败的用户研究 / 调研自动化案例     |
| LeadBAY    | 失败的 B2B 销售线索工具             |
| Juicebox   | 失败的 AI 招聘搜索 / 招聘自动化工具 |

### F3：融资很大但结果很差的公司

观察：

- 高融资后关停；
- 被低价收购；
- 裁员后转型；
- 用户增长过但收入没跟上；
- 估值很高但商业模式站不住。

### F4：反面信息源反推失败案例

来自 `radar/signals.md` 的反面信息源：

- 投诉 / 低分评论；
- 招聘岗位变化；
- 专利方向变化；
- 从业者匿名吐槽；
- 边缘玩家生存策略；
- 致命变量。

### F5：中国本土失败案例

优先补：

- 新消费品牌倒闭样本；
- 社区团购尾部公司；
- 在线教育转型失败样本；
- 低代码平台失败样本；
- 企业 SaaS 续费失败样本；
- 医美 / 口腔 / 皮肤管理信任崩坏样本；
- AI 教育工具失败样本。

### F6：仍活着但机制已经坏掉的准失败案例

状态可标：

- shrinking
- pivoted
- zombie
- disputed

不要等公司死透，关键是识别机制失效。

## 八、和产品扩容候选池的联动规则

每新增 3 个成功 / 机会候选，至少新增 1 个失败 / 反例候选。

A/B 级成功候选必须尽量配一个失败对照。

格式：

| 成功 / 机会候选 | 必配失败对照方向                    |
| --------------- | ----------------------------------- |
| ListenLabs      | 失败的用户研究 / 调研自动化案例     |
| LeadBAY         | 失败的 B2B 销售线索工具             |
| Juicebox        | 失败的 AI 招聘搜索 / 招聘自动化工具 |

------

## 九、同步规则

- 每月补仓时同时补失败案例；
- 失败案例不直接进入 case-cards；
- 只有完成轻量失败拆解或被成功案例反复引用后，才进入 case-cards；
- 失败案例必须能映射 `failure_modes.yml`；
- 不为失败而失败，不收无法抽象机制的八卦案例。
