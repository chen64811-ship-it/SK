# SKILL-入场协议设计:让用户先承认困境

> 版本:v1.0 · 2026-05-11
> 分类:术层 / Skill
> SK仓库用途:拆解 AI 产品时校准"用户进场动作";设计/评估新产品时检查"第一动作设计"是否做对;判断一个产品是焦虑驱动还是功能驱动。

---

## 一、为什么张雪峰那种生意收得动钱

张雪峰一次咨询收一两万,家长不眨眼。

你以为他在卖什么?填志愿的信息?这些信息官方系统全都免费,知乎上更新得比他还快。他真正卖的是另一个东西——

**"万一孩子选错专业,这不是我的错。我已经请了专家。"**

这是一张"决策免责权"。功能层面值 500 块,情感层面值 5000 块,存在层面值 20000 块——因为它替家长完成了"我尽到了父母责任"的自我赦免。

现在把这个机制放到 AI 产品上。同样是 AI 工具,为什么有的一上线就死,有的能让用户掏几百几千一年?

差异不在功能。差异在**进场协议**——用户第一次打开产品的那 30 秒,被你要求做了一个什么动作。

## 二、入场协议:三条透镜

判断一个 AI 产品有没有做对入场协议,用三条透镜并联检验:

### 透镜 1·入场不是功能,是焦虑命名

用户打开产品的第一眼,看到的不应该是"我们提供 XX 功能",而应该是**"你也有这种恐惧吗"**。

**对照例**:

- ❌ "用 AI 帮你写文案" → 用户问:"我什么时候需要这个?"(功能先行,情绪为零)
- ✅ "老板让你明天交方案,你现在只有标题" → 用户秒懂:"说的就是我"

**检验标准**:能不能用一句话让用户在 3 秒内觉得"这说的就是我"。如果需要解释才能让用户懂这个产品好在哪,大概率没做对入场。

### 透镜 2·第一动作是即时定心丸

焦虑被命名之后,产品必须立刻给用户一个**可执行的简单动作**,把"我不知道怎么办"切换成"我只需要做这一步"。

这个动作有三个硬指标:

| 要求 | 含义 |
|---|---|
| **低认知负担** | 不需要用户理解原理,不需要做选择题 |
| **5 分钟内反馈** | 用户付出的代价(一张照片/一段录音/一句描述)立即换回一个看得见的结果 |
| **可截图给别人看** | 结果有"我做了点什么"的可证明性——发给配偶/家长群/朋友圈都能用 |

张雪峰的"填表→出方案"就是这个标准动作。家长付完钱,几分钟后拿到一张写着孩子专业的清单,可以发给孩子妈、发给家庭群。**这个动作把"我不知道怎么办"翻译成了"我已经做了这件事"。**

### 透镜 3·有第二层

只有第一层的产品,是情绪套利,不是价值累积。

- 第一层(钩子):用确定性幻觉解决焦虑,完成付费转化
- 第二层(交付):让用户真实变好,建立复购和口碑

只有第一层的产品有三种必死的死法(详见 DOC-D023 伪需求陷阱):时间反噬(政策/共识窗口关闭)、结果反噬(用户后来发现没用,反噬指数级)、监管反噬(被认定为"加剧公共不公")。

**两层之间必须形成阶梯**,不能断裂:

```
第一层:即刻确定感(张雪峰模式 / MYHAIR 30 秒诊断)
    ↓
过渡层:过程可控感(里程碑 + 反馈)
    ↓
第二层:能力真实增长(结果可验证)
```

坏衔接的典型症状:第一层承诺"包你没事",第二层却说"还要努力"——用户感知"你骗我入坑,然后不管我了"。

正确衔接:第一层承诺"我帮你稳住",第二层承诺"我带你出去"——情绪承诺递进,而非断裂。

## 三、关键设计动作:让用户先"承认困境"

三条透镜里,**透镜 2 的"第一动作"是整个入场协议的胜负手**。

而第一动作的真正区分线,是一个被大多数产品设计者忽视的设计细节:

> **用户被要求做的第一个动作,是"承认困境",还是"接受功能"?**

### 两种动作的本质差异

**"承认困境"动作**:用户必须先在心里说一句"是的,我现在不行 / 我有这个问题",才能进入产品。

- MYHAIR AI:**拍一张你头顶的照片**(用户在心里承认:我可能有脱发)
- Yoodli:**上传你的演讲录像**(用户在心里承认:我演讲有问题)
- Spangle:**贴一个你想买的商品链接**(用户在心里承认:我已经心动了但没买/犹豫了)
- Lovable:**用一句话描述你想要什么 App**(用户在心里承认:我有个想法但做不出来)
- Gamma:**输入一句话告诉我你要做什么 PPT**(用户在心里承认:我要做但没时间做)

**"接受功能"动作**:用户被要求阅读、试用、配置——但不需要承认任何困境。

- 大多数 SaaS 工具站:看 dashboard、读文档、试 demo
- Notion 类知识管理:导入笔记、创建 workspace
- Cactus(餐饮 AI 电话接听):接入电话系统、配置话术

### 为什么"承认困境"是胜负手

**承认困境这个动作不是数据采集,是认知触发器。**

用户必须先在自己心里完成那次承认,定心丸才能在他身上起效。如果产品绕过这一步,即使后续功能很强,用户也很难产生"这个东西懂我"的感受——而这种感受是高客单价的唯一来源。

这也解释了一个 SK 库里的边缘观察:**Cactus 的电话接听是好工具,但不是焦虑解决器**。它接管的是显性问题(电话漏接),老板不需要承认"我接不动"。所以它能做生意,但很难做成高粘性、高溢价的产品——它只能卖工具的钱,不能卖免责权的钱。

### "承认困境"动作的设计清单

设计新产品或评估现有产品时,过这四条:

1. **承认的成本足够低**:一张照片、一句话、一个 URL——付出 30 秒以内
2. **承认的内容足够私密**:用户做这个动作时,身边没有人能看到(避免社交压力让他放弃)
3. **承认之后立刻有反馈**:不要让用户承认完然后等 24 小时,反馈窗口要在 5 分钟内
4. **承认的语言是"症状"不是"诊断"**:让用户描述他的处境(我经常加班/我头顶看起来这样/我演讲时紧张),不要让用户先给自己下诊断(我有焦虑症/我有 ADHD)——前者是开门动作,后者是关门动作

## 四、与仓库已有理论的关系

本 SKILL 不是新理论,是已有理论体系在"产品入场端"的操作化命名。

| 已有理论 | 关系 |
|---|---|
| **KANO·兴奋型需求**(DOC-S080) | KANO 讲了"什么是兴奋型",本 SKILL 讲"怎么让用户进场接收兴奋型"。是 KANO 的入场端补充 |
| **伪需求陷阱**(DOC-D023) | 伪需求陷阱讲"什么会死",本 SKILL 讲"怎么不死"——透镜 3"有第二层"就是反向防御:防止做出"假装在做"的单层产品 |
| **诊断空白四条件**(横向模式种子) | 四条件是医疗健康垂类的具体落地(高频焦虑/低频行动/诊断空白/B端导流),本 SKILL 是其上位抽象,适用所有 AI 产品类型 |
| **透过现象看本质**(DOC-S076) | S076 是认知方法,本 SKILL 是 S076 的产品化操作——把"看本质"翻译成"承认困境"这一个具体动作 |
| **JTBD(Jobs to be Done)** | JTBD 讲用户雇产品做什么任务,本 SKILL 在 JTBD 的"情感任务"层做了精确化:不是泛泛的"情感价值",而是"完成一次免责权交付" |

**调用顺序**:

```
拆解一个 AI 产品时:
  Step 1·先过伪需求三问(DOC-D023) → 是不是真需求
  Step 2·过本 SKILL 入场协议三透镜 → 入场设计对不对
  Step 3·过 KANO 三问(DOC-S080)   → 兴奋型还是期望型
  Step 4·若涉及医疗健康垂类 → 再过诊断空白四条件
```

## 五、在 AI 产品拆解中怎么用

按 product-teardown-template 的章节顺序,本 SKILL 在两处调用:

### 第 7 章·目标用户画像调用

在描述"目标用户是谁"之后,加一条入场协议描述:

> **入场协议**:用户进场时,产品要求的第一个动作是什么?这个动作是"承认困境"还是"接受功能"?用户在做这个动作时,内心承认了什么?

如果第一动作是"接受功能",说明产品做的是工具生意(可以赚钱但很难高粘性)。如果第一动作是"承认困境",说明产品做的是焦虑解决器生意(高粘性高溢价候选,但要继续看有没有第二层)。

### 第 10-A 章·中国机会扫描调用

判断"中国对标产品/复制可能性"时,用本 SKILL 做对比:

> 对标产品的入场动作是什么?中国市场上同赛道产品的入场动作是什么?差异在哪?如果中国市场的所有玩家都在做"接受功能"型入场,而对标产品做的是"承认困境"型入场,**这可能就是机会窗口的核心**。

### 三个检验问题(写完拆解后回头检验)

1. 这个产品的第一动作设计,用户付出的"承认成本"是多少?(秒数 + 心理代价)
2. 这个产品做完第一动作后,5 分钟内用户得到了什么反馈?这个反馈让用户从"焦虑"切换到了"我做了点什么"吗?
3. 如果剥离所有功能,只卖"焦虑缓解",有没有人买?(参照张雪峰检验)

## 六、设计/评估新产品时怎么用

### 入场动作设计的三个要求

设计新产品时,**先设计入场动作,再设计功能**。

不要从"我能提供 XX 功能"出发,要从"用户深夜会搜什么词"出发——

- 用户搜的那个词,就是焦虑命名(透镜 1)
- 用户搜完之后期待看到的第一屏内容,就是第一动作(透镜 2)
- 第一动作之后,用户开始期待"真的变好",就是第二层入口(透镜 3)

### 验证方法

观察用户使用产品的前 30 秒行为(可以录屏 + 事后访谈):

| 观察点 | 健康信号 | 警告信号 |
|---|---|---|
| 用户看到首屏后第一句话 | "诶,说的就是我" | "这是干嘛的?" |
| 用户被要求做的第一动作 | 主动承认了一个困境 | 阅读了一段功能介绍 |
| 用户做完第一动作后的表情 | 松了一口气 / 好奇结果 | 困惑 / 不耐烦 |
| 用户做完之后会不会截图 | 会,而且想发给某个人 | 不会 |

任何一条出现警告信号 → 入场协议要重新设计。

## 七、核心洞察金句

1. **"用户为'我懂你的恐惧'付费,而不是为功能付费。"**
2. **"承认困境的动作不是数据采集,是认知触发器——用户必须先在心里承认,定心丸才能在他身上起效。"**
3. **"第一动作的真正区分线:用户被要求'承认困境'还是'接受功能'?前者是焦虑解决器,后者是工具。"**
4. **"只有第一层的产品是情绪套利,不是价值累积——三种死法都跑不掉。"**
5. **"先设计入场动作,再设计功能。从'用户深夜会搜什么词'出发,不是从'我能提供什么'出发。"**

## 八、在仓库中的位置

**归属判断**:术层(Skill),不是道层(Theory)。

理由:本 SKILL 是一个可操作的判断工具 + 设计清单,不是世界观级的根本理论。世界观级的判断由 KANO、伪需求陷阱、破坏式创新、从0到1 等 D/S 类道层文档承担。

**触发场景**:

- 拆解 AI 产品时,做 product-teardown-template 第 7 章"目标用户画像"或第 10-A 章"中国机会扫描" → 调用本文
- 设计/评估新产品时,判断"第一动作设计"是否做对 → 调用本文
- 推演新方向时,判断这个方向是"焦虑驱动"还是"功能驱动" → 调用本文
- 看到一个产品做得好但说不清为什么 → 调用本文第三节"承认困境 vs 接受功能"做对照

**与本文相关的文档**:

- *DOC-S080(KANO需求分层)—— 兴奋型的入场端补充*
- *DOC-D023(伪需求陷阱)—— 反向防御,防止做出单层情绪套利产品*
- *横向模式种子-AI诊断市场通用公式 —— 本 SKILL 在医疗健康垂类的具体落地*
- *DOC-S076(透过现象看本质)—— 本 SKILL 是其产品化操作*
- *core/项目审问清单 —— Part C 正向三问中,本 SKILL 是"是否真做对了入场"的补充透镜*

## 九、本 SKILL 来源

2026-05-11 对话中,从一段关于"张雪峰为什么能收钱"和"焦虑解决器先于功能"的讨论中提炼。

讨论原始内容指向一个产品入场协议的雏形(焦虑命名 + 即时定心丸 + 第二层)。在用 SK 库 38 个 AI 产品对标库扫描后,发现"让用户先承认困境"是 13 个"真做对了焦虑解决器"产品的共同动作——这一发现在原讨论中没有被显式命名,经扫描后提炼为本 SKILL 的胜负手设计。
