## 小王的核心定位

小王现在最重要的不是“发文章的人”，而是：

> **读者视角的产品评审者 + 需求链接者。**

仓库里对小王的定位已经很清楚：他有双角色，一是文章流水线执行者，二是产品需求链接者和全品评审者。尤其第二个角色更重要，因为小王掌握评论、私信、跨平台反馈，更接近真实读者视角。

所以小王以后主要做五件事：

### 1. 内容线保底

继续负责：

- 底稿标注；
- GPTS 改写；
- 第一读者检测；
- 排版发布；
- 冷启动；
- 分发；
- 评论/私信/朋友圈/跨平台反馈捕捞。

这条线不能断。
 它是 SK 对外释放信号的主线。

### 2. 需求澄清

当杨提出一个产品方向，小王要问：

- 那个卡住的人是谁？
- 他为什么卡住？
- 这个产品一句话怎么解释？
- 如果写一篇文章介绍它，读者能不能听懂？
- 这个需求是真用户需求，还是我们推理出来的伪需求？

这就是小王作为产品经理实习生的核心训练。

### 3. 产品评审备忘

每次产品讨论后，小王 24 小时内写：

> 《产品需求评审备忘》

内容包括：

- 本次共识；
- 我听懂的部分；
- 我没听懂的部分；
- 我作为评审者的 3 个质疑；
- 下一步杨、小王、小陈分别做什么。

这个文件非常关键。
 它不是会议纪要，而是防止三个人理解错位。

### 4. MVP 验收视角

小陈做出工具或 MVP 后，小王不是看代码，而是看：

- 这个东西用户第一步知道怎么用吗？
- 输出结果能不能看懂？
- 它解决的问题能不能写成文章？
- 如果发出去，能不能钓到真实用户？
- 有没有偏离最初的“那个卡住的人”？

### 5. 小陈的协作桥

小王每周和小陈做一次 15 分钟同步：

- 这周做了什么？
- 卡在哪里？
- 哪个地方需要杨拍板？
- 有没有技术上跑偏？
- 有没有需求没说清楚？

仓库里原本也明确：王不是技术审查者，而是技术实习生的链接者和减压阀，周做得顺就不打扰，迷路了先帮他找回来，找不回来再找杨。

## 小陈的核心定位

小陈的核心定位应该是：

> **技术实现者 + Agent 工具链建设者 + MVP 工程执行者。**

但要特别强调：
 他不是“杨说什么就写什么”的代码工人。

原来实习生 B 手册里有一句很重要的话：

> 动手之前必须先搞懂产品构思、卡住的人、对标逻辑，再做最简 MVP。

小陈以后主要做六件事。

### 1. 先搞懂，再动手

每个方向开始前，小陈必须先搞懂：

- 杨为什么选这个方向；
- 对标产品做对了什么；
- 目标用户是谁；
- 用户卡在哪一步；
- 最简 MVP 到底验证什么；
- 哪些地方必须自动化，哪些可以人工先顶着。

这点要写死。
 如果他一上来就写代码，容易做出一个“技术上能跑、产品上没用”的东西。

### 2. 对标产品技术拆解

每个产品方向启动前，小陈要做一份：

> 《对标产品技术拆解》

内容包括：

- 用户端看到什么；
- 用户完成核心动作需要几步；
- 推测技术实现；
- 我们最简 MVP 怎么做；
- 哪些用现成工具；
- 哪些必须写代码；
- 哪些可以先人工；
- 我的疑问。

这对应仓库原来对技术实习生的要求：动手前先做对标产品技术拆解，而不是直接开干。

### 3. SK 工具链半自动化

这是小陈最适合补上的新缺口。

他可以做：

- 发布前 QA 检查器；
- 产品拆解字段提取器；
- 失败模式对照检查器；
- 案例卡结构化工具；
- Hermes / Claude Code 可执行任务模板；
- SK 文档路由器；
- 文章输入 → 检查报告输出的小工具。

这也符合招聘工具包里对 AI 应用实习生的定义：不是从零搭系统，而是把已经写好的纸面流程变成脚本、工具和可运行系统。

### 4. 最简 MVP 实现

当某个产品方向确定后，小陈负责做最简 MVP。

这里的关键是：

> 最简 MVP 不是功能少，而是能让一个真实用户跑完一次核心流程。

比如：

- 不一定要登录系统；
- 不一定要完整后台；
- 不一定要自动化全部流程；
- 前 10 个用户可以人工 + AI 半自动处理；
- 目标是拿到真实反馈，而不是做一个漂亮系统。

原来的 B 手册也强调：能不用代码就不用代码，能用现成工具就用现成工具，前 10 个用户可以先人工处理。

### 5. 技术可行性判断

小陈应该拥有技术可行性判断权。

他要能直接说：

- 这个功能 1 天能做；
- 这个功能要 2 周；
- 这个功能现在没必要；
- 这个功能可以用现成工具替代；
- 这个功能技术上能做，但不适合第一版。

这就是他作为“技术架构 + Agent 工程”的价值。

### 6. 用 AI 编程，但不外包判断

小陈可以 all in Claude Code、Copilot、MiniMax。
 但他必须做到：

- 先拆任务；
- 再让 AI 写；
- 自己检查；
- 自己运行；
- 自己调试；
- 自己能讲清楚代码为什么这样写。

这和我们之前面试设计里说的一样：
 AI 是放大器，不是替代判断。

- 检查有没有偏离需求；
- 整理卡点给杨；
- 后续帮忙做用户测试话术和反馈整理。

杨做：

- 只在关键节点拍板；
- 不陷入具体代码；
- 不把所有沟通都抓在自己手里。

# 二、招募不是 HR 工作，是产品工作

你和小王要一起努力的不是“写招聘文案”，而是：

> **把一个模糊项目，解释成一个具体的人愿意靠近的机会。**

这件事很难。

因为 SK 不是普通公司，不是标准岗位，也不是一个已经定型的产品。

所以招募文必须回答清楚：

1. 我们到底在做什么？
2. 为什么现在需要你？
3. 你来了第一周做什么？
4. 你短期能得到什么？
5. 长期可能走向哪里？
6. 什么人不适合？
7. 怎么判断双方是否同频？

仓库里的招聘工具包其实已经有这个方向：用仓库做前置过滤器，让不合适的人自己离开，让真正感兴趣的人读完再来；它也强调主任务不是从零搭系统，而是把已有工作流变成自动化工具。

但现在你们要再往前走一步：

> **不只是让人自己来，还要主动把这套机会讲给对的人听。**

4. 长期可能走向哪里；
5. 不能接受什么人；
6. 最终是否录用。

也就是说：

```
你负责定“找什么人、为什么找、标准是什么”。
小王负责帮你把它写清楚、问清楚、反馈回来。
```

你给方向，小王做翻译和校正。

# 五、主动出击应该怎么做

主动出击不是“到处发招聘”。

主动出击应该是：

> **带着一篇清楚的招募文，去找已经可能同频的人。**

可以分三类。

## 第一类：技术型候选人

目标：

- AI 应用开发；
- Python / FastAPI；
- Claude Code / Cursor 用户；
- RAG / Agent 项目经验；
- 研二、研三、应届生；
- 长沙本地优先。

主动出击话术不能一上来讲人生方法论。

要讲：

> 我们不是招普通后端，是招能把 AI 工作流做成工具的人。
>  仓库公开，你先看 README。
>  如果你对 Agent / MVP / 自动化工作流感兴趣，我们聊。

------

## 第二类：行业专家 / 真实用户

目标：

- 医疗慢病管理；
- 教培；
- 金融销售训练；
- 出海；
- 企业服务；
- 你正在推演的赛道中的从业者。

这类不是招实习生，而是招募“共创访谈对象”。

话术要讲：

> 我们正在用 AI 产品拆解的方法研究这个方向，不是来卖东西。
>  想找真实从业者帮我们校准判断。
>  如果你愿意聊 30 分钟，我们会把研究结论整理给你。

------

## 第三类：长期同路人

目标：

- 小团队创业者；
- 产品经理；
- 技术合伙型人才；
- 独立开发者；
- 读完文章后主动加你的人。

这类不能急着招。

要先通过文章、仓库、一次深聊、一次小任务来判断。

话术要讲：

> 不是现在就谈合伙，而是先一起做一件小事，看判断方式是否同频。

# 六、招募文以后应该分成 4 种

不要所有招募都写成一篇。

以后应该有四种标准模板：

## 1. 实习生招募文

对象：小陈这种人。

核心问题：

> 你来了第一周做什么？
>  你能学到什么？
>  为什么这不是普通实习？

------

## 2. 技术伙伴招募文

对象：更成熟一点的 AI 工程师 / 独立开发者。

核心问题：

> 我们要做什么 MVP？
>  技术难点是什么？
>  合作边界是什么？

------

## 3. 行业共创招募文

对象：医生、老师、金融销售、出海从业者等。

核心问题：

> 我们需要你提供什么真实经验？
>  你能得到什么研究反馈？
>  为什么不是找你买服务或卖服务？

------

## 4. 用户测试招募文

对象：真实卡住的人。

核心问题：

> 你是不是那个卡住的人？
>  你现在遇到什么问题？
>  我们能帮你试一次什么？
>  你需要付出什么？
>  你会得到什么？

------
