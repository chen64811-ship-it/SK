## 一、一句话分工

```text
杨：方向判断 + 产品推演 + 最终拍板

实习生A（产品经理实习生 + 架构助理）
负责：内容流水线、读者反馈捕捞、需求澄清、产品评审、招募表达协作、工具验收

实习生B（AI Agent / 工具链 / MVP 工程实习生）
负责：技术实现、Agent 工具链、SK Workbench、对标技术拆解、最简 MVP
```

A 不是 B 的上级。B 也不是 A 的外包开发。

**A 帮 B 不做偏；B 帮 A 把想法做成能跑的工具。**

## 二、四条工作线

原有三条对外信号线保留，同时新增一条内部工具线。

```text
线1：内容线
文章发布、分发、评论捕捞、读者信号。

线2：产品线
产品推演、对标拆解、MVP、用户反馈。

线3：招募线
实习生、技术伙伴、行业专家、真实用户、长期同路人的招募与主动出击。

线4：工具线 / SK Workbench 线
把 GPTS、SK 工作台、Claude Projects、Claude Code、Hermes 中已经验证有效的动作，沉淀成我们自己的 SK 工具系统。
```

**重要判断：**

GPTS、SK 工作台、Claude Projects 是原型工具，不是最终系统。  
最终应逐步沉淀为 **SK Workbench**：以 SKILL 为核心、以 GitHub 为事实源、以多模型为引擎、以人工判断为闸门的内部工具系统。  
主流程应逐步变成：

```text
任务 → 路由到 SKILL → 加载必读文件 → 调用模型 → 输出结构化结果 → 人审核 → 入库
```

### 5. Hermes

定位：仓库自动化执行者。

给新人理解的解释：

Hermes 不是普通聊天工具，而是未来可能用来「按明确指令修改仓库、生成文件、检查字段、执行固定维护任务」的 Agent 执行层。第一月 B 只需要理解它的作用和风险，不需要真正操作 Hermes。

杨需要在第一月内给 B 看一次 Hermes 的文档或演示，让 B 知道：

- Hermes 能做什么；
- Hermes 不能做什么；
- 为什么不能让 Hermes 自动改核心文件。

使用时机：

- 第一月只观察和学习；
- 第二月以后可尝试边界明确的小任务；
- 例如生成固定格式文件、检查缺字段、整理工具运行结果。

禁止：

- 不允许自动修改核心判断类文件；
- 不允许自动改 README、北极星、执行状态总表、系统指令；
- 不允许绕过杨审核直接写入主仓库。

## 四、实习生A 核心定位

> 读者视角的产品评审者 + 需求链接者 + 招募表达协作者 + 工具验收者

A 掌握读者原话、评论、私信、朋友圈、跨平台反馈，比杨和 B 更接近真实用户视角。这是 A 最不可替代的能力。

### A 每周固定六件事

1. **内容线保底**  
   底稿标注 → GPTS 改写 → 第一读者检测 → 排版发布 → 冷启动 → 分发 → 反馈捕捞。

2. **需求澄清**  
   每次产品讨论后，追问：
   - 那个卡住的人是谁？
   - 他为什么卡住？
   - 这个产品一句话怎么解释？
   - 读者能不能复述？

3. **产品评审备忘**  
   产品讨论后 24 小时内产出《产品需求评审备忘》：
   - 本次共识；
   - 我听懂的；
   - 我没听懂的；
   - 我的 3 个质疑；
   - 下一步各自做什么。

4. **MVP / 工具验收视角**  
   B 交付工具后，A 不看代码，只看：
   - 用户第一步知道怎么用吗？
   - 输出能不能看懂？
   - 能不能帮助真实工作流？
   - 能不能写成文章或招募文钓到真实用户？

5. **招募线协作**  
   每周迭代一版招募私信文案；
   维护招募追踪表；
   记录候选人最大疑问；
   面试后写观察复盘。

6. **SKILL 使用与反馈**  
   A 是 Article QA Skill Runner、Recruit Message Runner 的主要使用者和验收者。
   A 不需要写代码，但必须能判断工具输出是否真的减少了工作成本。

### A 不做什么

- 不替 B 选技术栈；
- 不评价代码好不好；
- 不替杨做最终方向判断；
- 不自己动手做 MVP；
- 不把工具输出当成最终事实。

## 五、实习生B 核心定位

> 技术实现者 + Agent 工具链建设者 + SK Workbench 工程执行者 + 最简 MVP 实现者

B 不是“杨说什么就写什么”的代码工人。

**动手之前必须先搞懂产品构思、卡住的人、对标逻辑、工作流痛点，再做最小工具。**

### B 每周固定七件事

1. **先搞懂再动手**  
   每个方向开始前，必须能回答：
   - 杨为什么选这个方向？
   - 目标用户卡在哪一步？
   - 最简 MVP 验证什么？
   - A 为什么觉得这个地方读者会卡住？

2. **对标产品技术拆解**  
   每个产品方向启动前，产出《对标产品技术拆解》：
   - 用户端看到什么；
   - 核心动作几步；
   - 推测技术实现；
   - 我们最简 MVP 怎么做；
   - 哪些用现成工具；
   - 哪些必须写代码；
   - 哪些先人工。

3. **SK Workbench 建设**  
   把 GPTS、SK 工作台、Claude Projects 中已经验证有效的动作，做成自己的可运行工具。

4. **SKILL Runner 实现**  
   每个工具都不是孤立脚本，而是一个 SKILL：
   - skill 配置；
   - prompt 模板；
   - required_files；
   - output_schema；
   - validator；
   - writer。

5. **最简 MVP 实现**  
   能让真实用户跑完一次核心流程即合格，不追求完整后台。

6. **技术可行性判断**  
   B 有权直接说：
   - 这个功能 1 天能做；
   - 这个可以用现成工具替代；
   - 这个第一版不适合做；
   - 这个技术上能做但不值得现在做。

7. **用 AI 编程但不外包判断**  
   可以 all in Claude Code / Cursor / Copilot，但必须自己能解释：
   - 为什么这样拆；
   - 为什么这样写；
   - 怎么测试；
   - 哪些地方不确定。

### B 不做什么

- 不替杨选方向；
- 不绕过 A 直接按自己理解做大系统；
- 不一上来做全自动平台；
- 不把 SK 理解成普通 RAG 知识库；
- 不把“能跑”当成“做完”；
- 不直接自动写入核心仓库文件。

## 七、协作流程（三段式）

### 阶段1：方向进入前

| 人 | 做什么 |
|----|--------|
| A | 读推演流水账 + 读案例卡 + 提出用户视角问题 + 判断这个方向能不能写成读者听得懂的文章 |
| B | 读对标产品资料 + 试用产品 + 做技术拆解 + 判断最简 MVP 或工具能不能做 |
| 共同产出 | 产品讨论会前准备：A 的用户/需求疑问 + B 的技术/流程疑问 |

---

### 阶段2：产品讨论会

| 人 | 负责问 |
|----|--------|
| A | 用户是谁？文章怎么讲？读者能不能复述？需求哪里模糊？ |
| B | 第一版输入输出是什么？哪些能自动化？需要哪些 API / 工具？技术难点是什么？ |

会后 24 小时内：

- A 交付：《产品需求评审备忘》；
- B 交付：《MVP 技术拆解》或《SKILL Runner 技术方案》；
- 杨：最终取舍与下一步拍板。

---

### 阶段3：开发或工具化中

| 人 | 做什么 |
|----|--------|
| B | 搭工具 / 写代码 / 接 API / 跑测试 / 写 README / 每周发进度；**目标是把可复用动作做成 SKILL Runner**，而不是堆孤立脚本 |
| A | 每周 15 分钟同步；**不看代码**，从真实工作流视角验收输出是否可用、是否减少重复与跑偏；整理卡点给杨 |
| 杨 | 只在关键节点拍板，不陷入具体代码；**判断该工具是否值得进入 SK 主流程**（是否入库、是否推广到默认工作流） |

**本阶段补充原则：**

- B 不只是写代码，而是把工具做成 **SKILL Runner**（配置、必读文件、输出结构、验收）。
- A 不评价代码好坏，只验收 **输出能否进入 A 的真实工作流**。
- 杨判断 **是否进入下一版 / 是否进入 SK 主流程**，避免工具链空转。

## 八、招募分工

| 人 | 做什么 |
|----|--------|
| 杨 | 判断现在招哪类人 / 定义候选人画像 / 写第一版核心意图 / 决定是否进入面试 / 负责关键候选人的深聊 |

| B | 未来参与技术类招募材料中的真实项目说明，例如 SK Workbench、Article QA Runner、MVP 工具等 |

招募文四种类型：

1. 实习生招募文；
2. 技术伙伴招募文；
3. 行业共创招募文；
4. 用户测试招募文。

### 八·补 主动出击三类人（v4.2 补回，来源：早期草稿）

主动出击不是到处发招聘，而是带着一篇清楚的招募文，去找已经可能同频的人。分三类：

**第一类：技术型候选人**

- 目标：AI 应用开发 / Python / FastAPI / Claude Code 用户 / RAG / Agent 项目经验 / 研二、研三、应届生 / 长沙本地优先
- 接近话术（不讲人生方法论）：

> 我们不是招普通后端，是招能把 AI 工作流做成工具的人。
> 仓库公开，你先看 README。
> 如果你对 Agent / MVP / 自动化工作流感兴趣，我们聊。

**第二类：行业专家 / 真实用户**

- 目标：医疗慢病管理 / 教培 / 金融销售训练 / 出海 / 企业服务 / 当前正在推演的赛道里的从业者
- 这类不是招实习生，而是招募"共创访谈对象"
- 接近话术：

> 我们正在用 AI 产品拆解的方法研究这个方向，不是来卖东西。
> 想找真实从业者帮我们校准判断。
> 如果你愿意聊 30 分钟，我们会把研究结论整理给你。

**第三类：长期同路人**

## 十、常见失误预警

### A 最容易犯的错

- 把“没收到反馈”误认为“没有 A 档读者”；
- 产品评审备忘写成会议纪要，没有质疑；
- 招募文写得像招聘启事，不像真实邀请；
- 验收工具时说“还不错”，但没指出具体能不能用；
- 把 GPT 输出直接当成可发布稿。

### B 最容易犯的错

- 把 Claude Code 的速度误认为自己的工程判断已经成熟；
- 熟悉 Claude Code 后，忍不住直接写大系统；
- 把 SK Workbench 做成低配 ChatGPT；
- 把 RAG 当成万能入口；
- 只做功能，不做 skill 配置、输出格式和验收；
- 遇到问题沉默超过 30 分钟；
- 把“能跑”当成“完成”。

## 十一、30 天成功标准

30 天后回看，至少要完成：

1. 内容线没有断，文章发布流程继续跑；
2. A 完成至少 1 份产品需求评审备忘；
3. A 完成至少 2 版招募私信文案迭代，并记录反馈；
4. B 跑通 Article QA Skill Runner v0.1；
5. A 用 Article QA Skill Runner 验收过至少 5 篇测试稿；
6. B 完成 Recruit Message Runner 或 Repo Audit Runner 的 v0.1 方案；
7. B 能讲清楚 SK Workbench 的目录、skill 配置、模型调用、输出验收；
8. 杨能判断：这个工具链是否真的减少了 SK 的重复动作和跑偏风险。

## 十二、核心原则

```text
杨：判断系统的大脑
A：判断系统和读者/候选人之间的翻译器
B：判断系统变成工具和 MVP 的工程手
```

最终目标不是“多用几个 AI 工具”，也不是把 SK Workbench 做成低配 ChatGPT 或完整 RAG 平台，而是：

```text
让 SK 从“人脑 + 文档 + GPT 对话”
逐步变成
“人判断为核心，SKILL 和 Agent 工具链辅助执行的产品探索系统”。
```
